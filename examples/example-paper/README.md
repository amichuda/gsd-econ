# Example: a worked phase

This directory shows what the planning artifacts look like for one phase of a hypothetical paper. It's a documentation walkthrough, not a runnable project.

The hypothetical paper studies **the effect of mobile-money rollout on remittance-driven consumption smoothing in rural Uganda**. It uses a staggered DiD design exploiting district-level variation in mobile-money agent deployment between 2008 and 2014.

## Files in this directory

- `PROJECT.md` — top-level project description
- `REQUIREMENTS.md` — hypotheses, primary outcomes, sample, locked vs open spec choices
- `METHODOLOGY.md` — the methodology declaration that drives test loading
- `ROADMAP.md` — phased pipeline customized for this paper

## How to use this example

1. Read `METHODOLOGY.md` first — it shows how the methodology declaration drives the rest of the workflow. Note `primary: did`, `secondary: [ols]`, `prereg.required: false` (this is observational, not RCT).

2. Read `REQUIREMENTS.md` — see how decisions are split between LOCKED (committed) and OPEN (still up for grabs). Note that the staggered estimator choice is OPEN at project bootstrap, to be locked in Phase 3 (identification diagnostics).

3. Read `ROADMAP.md` — adapted for this paper. Phase 6 (Monte Carlo) is dropped (not a methods paper); a Phase 3a is added for handling pre-treatment trends in the rollout.

4. `PROJECT.md` shows the project-level metadata.

## What's NOT in this example

- No code (this is documentation, not a runnable analysis)
- No data (would be restricted-access UCC mobile-money rollout data + UBOS household surveys)
- No actual phase outputs (CONTEXT.md, PLAN.md, etc.) — these would be generated by running the gsd-econ commands

## Running this example for real

In a real workflow, you'd:

```bash
# Initialize
mkdir uganda-mobile-money && cd uganda-mobile-money
git init
npx get-shit-done-cc@latest --opencode --local --minimal
git submodule add https://github.com/<you>/gsd-econ vendor/gsd-econ
git submodule add https://github.com/rdahis/research-unit-tests vendor/research-unit-tests
bash vendor/gsd-econ/scripts/install.sh

# In your runtime
/gsd-new-paper
# (work through the framing interview)
# (literature scout runs)
# (PROJECT/REQUIREMENTS/METHODOLOGY/ROADMAP populated)

/gsd-discuss-identification 1
# (data-cleaning phase; light identification touch)

/gsd-plan-empirics 1
/gsd-execute-phase 1
/gsd-verify-replication 1

/gsd-discuss-identification 2
# ...and so on through to Phase 9 (submission)
```

By Phase 3 (identification diagnostics) you're locking the staggered DiD estimator choice. By Phase 7 you're running `/gsd-tables-figures`. By Phase 9 you're running `/gsd-submit-paper`, which invokes `/gsd-referee-sim` and packages the bundle.

## Why this example?

It's a reasonable analog to actual labor/development empirical work — staggered rollout, observational identification, multiple secondary outcomes, no pre-registration but discipline expected. It exercises the DiD-relevant tests, the universal tests, and (because it's spatial) the Conley SE warning.

It's also intentionally not a copy of any specific real paper.
